# TypeCarrier 历史清理与保留策略设计

状态：后续设计。本文只覆盖轻量维护能力，不把 TypeCarrier 扩展成剪贴板历史管理器。

## 目标

- 让草稿、发送历史和 Mac 接收历史不会无限增长。
- 给用户必要的清理控制：批量删除、保留期限、条目上限。
- 保持历史功能服务于“重新发送、排查、找回最近文本”，不发展成内容管理系统。

## 非目标

- 不做收藏、置顶、重命名、标签、文件夹或复杂归档。
- 不搜索或保存系统剪贴板历史。
- 不做跨设备历史同步。
- 不做富文本、图片、文件或链接管理。

## 范围

### 条目上限

- 草稿、发送历史、Mac 接收历史分别有上限。
- 默认上限保持保守，优先减少本地持久化和隐私风险。
- 超过上限时按更新时间保留最近条目。

### 保留期限

候选设置：

- 7 天
- 30 天
- 90 天
- 永久，受条目上限约束

第一版可以只在设置中提供一个全局保留期限，后续再考虑草稿和历史分开配置。

### 批量删除

- 支持清空当前列表。
- 支持选择多条删除。
- 删除前给出数量确认。
- 删除失败时保留可见错误，不静默吞掉。

### 搜索关系

搜索功能只搜索保留下来的 TypeCarrier 记录。被保留期限或条目上限清理掉的记录不再出现在搜索结果中。

## 数据和隐私

- 历史仍保存在本地应用数据目录。
- 设置变更后不需要立刻删除；可以在下次写入、启动或用户点击“立即清理”时执行 prune。
- 诊断导出默认不应包含完整历史文本，除非用户明确导出带内容的排查包。

## UI 建议

- 历史页保留当前清空入口，并增加多选删除。
- 设置页增加“历史保留”分组：保留期限、条目上限、立即清理。
- 空状态和清理结果只说清数量，不引入复杂统计面板。

## 测试计划

### Core Tests

- 超过条目上限时保留最近记录。
- 超过保留期限的记录会被清理。
- 草稿和历史删除不会互相误删。
- 批量删除只删除选中的 id。

### iOS Validation

- 历史和草稿能批量删除。
- 修改保留期限后，列表在下一次清理点正确更新。
- 删除失败时，用户能看到明确错误。

### macOS Validation

- 接收历史遵守条目上限和保留期限。
- 删除接收历史不影响连接诊断日志。
- 历史清理不影响下一次接收和粘贴。

## 建议拆分

1. 先扩展 `CarrierRecordStore` 的 prune 策略，支持条目上限和保留期限。
2. 再补 Core 测试覆盖上限、期限和批量删除。
3. 再做 iOS 历史页多选删除和设置入口。
4. 最后做 macOS 接收历史的同等清理能力。
