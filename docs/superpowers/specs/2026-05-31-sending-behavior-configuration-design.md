# TypeCarrier 发送行为配置设计

状态：后续设计。本文整理已经陆续出现的发送行为能力，不代表当前版本已经完整支持这些配置。

## 目标

- 把 TypeCarrier 的核心手感整理成少量可解释的配置。
- 让用户清楚知道文本发送后，iPhone 输入区、Mac 剪贴板、Mac 当前输入焦点会发生什么。
- 失败时优先保住用户输入内容和恢复路径，避免“看起来成功但实际没有输入”的误判。

## 非目标

- 不把 TypeCarrier 做成剪贴板管理器。
- 不支持按 App、按设备、按文本类型配置复杂规则。
- 不用发送配置绕过 macOS Accessibility、受保护输入或目标 App 拒绝粘贴的限制。

## 配置范围

### iOS 发送后编辑器行为

第一版建议提供三个选项：

- 成功后清空：适合连续输入多段短文本，作为默认值。
- 成功后保留：适合反复修改同一段文本。
- 成功后全选：适合发送后立刻覆盖重写。

失败、未收到 Mac 回执、Mac 粘贴失败或状态不可验证时，应保留当前文本，不自动清空。

### Mac 接收后插入行为

第一版建议提供两个模式：

- 自动粘贴到当前焦点：默认模式，保持 TypeCarrier 的核心差异化。
- 仅复制到 Mac 剪贴板：作为 fallback，用于目标 App 不接受模拟粘贴、用户不确定当前焦点、或不想让 Mac 自动输入的场景。

确认后粘贴可以作为后续模式：Mac 收到文本后先在菜单栏或窗口中提示，由用户确认后再粘贴。它会增加一步操作，不适合作为默认路径。

### Mac 剪贴板恢复

- 默认尽量恢复原剪贴板内容。
- 如果恢复失败，或剪贴板在发送后被用户/其他 App 改过，不应覆盖新内容。
- 仅复制到剪贴板模式下不恢复，因为用户的明确意图就是把内容放到 Mac 剪贴板。

## 状态和回执语义

- iOS 本地保存成功只代表文本已进入本地历史，不代表 Mac 已输入。
- Mac 收到 payload 只代表传输成功，不代表粘贴成功。
- Mac 发出 Command-V 也不能单独代表成功；如果目标不可验证或明确拒绝，应返回失败或不可验证状态。
- iOS 只有在 Mac 返回明确可接受的成功回执后，才按“成功后编辑器行为”处理输入区。

## UI 建议

- iOS 设置中放“发送后处理”：清空、保留、全选。
- Mac 设置中放“接收后处理”：自动粘贴、仅复制到剪贴板、未来可扩展确认后粘贴。
- Mac 设置中放“剪贴板恢复”：默认开启；仅在自动粘贴模式下生效。
- 每个设置项用一行短说明解释影响，不在主发送界面增加额外负担。

## 错误处理

- Mac 无 Accessibility 权限：保留 iOS 文本，提示到 Mac 授权。
- Mac 当前焦点不可验证：保留 iOS 文本，提示用户确认焦点或手动粘贴剪贴板内容。
- Mac 写剪贴板失败：保留 iOS 文本，提示重试或检查剪贴板工具。
- Mac 已复制但未粘贴：iOS 不应显示为“已输入”，而应显示“已复制到 Mac 剪贴板”。

## 测试计划

### Core Tests

- 不同 paste status 下，编辑器清空策略符合配置。
- 失败和不可验证状态不会清空 iOS 文本。
- 仅复制到剪贴板模式产生独立状态，不混同自动粘贴成功。

### iOS Validation

- 成功后清空、保留、全选三个模式都能按预期工作。
- Mac 粘贴失败时，iOS 文本保留且状态明确。
- 从草稿或历史重新发送时，也遵守同一套行为配置。

### macOS Validation

- 自动粘贴模式继续走当前剪贴板写入、Command-V 和恢复流程。
- 仅复制到剪贴板模式不发送 Command-V，也不恢复剪贴板。
- Accessibility 未授权时，自动粘贴模式给出明确失败；仅复制到剪贴板模式不要求 Accessibility。

## 建议拆分

1. 先补 Core 层的发送后编辑器策略测试，锁定 receipt 到编辑器行为的映射。
2. 再抽出 Mac 接收后处理模式，让自动粘贴和仅复制到剪贴板分支清晰。
3. 再做 iOS / macOS 设置 UI 和持久化。
4. 最后更新 release notes、验证清单和故障提示文案。
